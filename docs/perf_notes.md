# Замеры производительности ML-sidecar (ТЗ v2.1 §2)

Данные для решения о деплое (§6.2): хватит ли минимального инстанса без GPU.
Скрипт: `ml/scripts/measure_perf.py` (CPU-инференс, torch CPU-build).

## Результаты (15.07.2026, torch 2.x CPU, Python 3.14)

| Метрика | Значение (CPU) |
|---|---|
| RSS до загрузки моделей | 15 MB |
| Загрузка reranker (mmarco-mMiniLMv2-L12-H384-v1) | 9.1 с, +718 MB |
| Загрузка guard-kg (XLM-R ragguard-kyrgyz-v1) | 2.2 с, +489 MB |
| RSS после обеих моделей + инференс | ~1550 MB |
| /rerank: 20 кандидатов | **182 ms** |
| /guard/check: 1 пара | **53 ms** |

## Выводы для деплоя (§6.2)

- Латентность CPU отличная: rerank+guard добавляют ~250 мс к пайплайну /ask — незаметно на фоне LLM-генерации (секунды).
- **RAM — узкое место: ~1.5 ГБ с GUARD_MODELS=kg.** Инстансы 512 MB–1 GB не подходят.
  Варианты: (а) инстанс ≥2 GB (~$10–15/мес на Railway/Render), (б) ONNX int8-квантизация
  XLM-R (~4× меньше RAM на модель) + возможно дистилляция reranker, (в) GUARD_MODELS=kg
  уже включён (en-модель не грузится — экономия ~0.5 ГБ).
- Загрузка моделей 11 с суммарно — lazy-загрузка при первом запросе допустима,
  но на проде лучше прогреть перекрёстным запросом после старта (cold start иначе ударит по первому пользователю).

## Методика

- RSS процесса до/после загрузки каждой модели;
- латентность: среднее из 5 прогонов после 1 прогрева;
- /rerank — 20 пар (вопрос × кандидат ~600 символов), /guard — 1 пара;
- машина: Windows 11, CPU-режим (без CUDA) — сознательно, это профиль прод-инстанса.
