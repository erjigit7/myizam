# «Мыйзам» — Спецификация Ingestion под реальное API cbd.minjust.gov.kg

Дополнение к ТЗ v2.0: заменяет §4.1–4.4. Основано на результатах разведки API (июль 2026).

---

## 1. Итоги разведки (зафиксированные факты)

| Факт | Следствие для Ingestion |
|---|---|
| Все данные доступны через JSON API без авторизации | Обычный HttpClient; Playwright НЕ нужен |
| `GetEdition` отдаёт документ как единую HTML-строку (экспорт из Word) в поле `contentRu` | Структуру (Раздел/Глава/Статья) вычленяем сами по текстовым маркерам |
| Статус и список редакций — в отдельном `GetDocument` | Двухшаговый пайплайн: сначала метаданные и проверки, потом текст |
| Кыргызский текст приходит в том же поле `contentRu` при `lang=kg`; поле `contentKg` — всегда null (легаси) | Язык определяется ПАРАМЕТРОМ запроса, не полем ответа. Задокументировать жирно — источник будущих багов |
| `GetEdition.dateOfFutureEntry` существует | Редакция может быть опубликована, но ещё не вступить в силу — фильтровать |
| Поиск `GetDocuments` нечёткий по умолчанию (`isPercentage:true`), в выдаче есть «Утратил силу» | Для MVP поиск НЕ автоматизируем: 7 documentCode подбираются руками один раз |
| У Трудового кодекса 2025 г. уже 6 редакций, свежая — 23.06.2026 | Законы правятся часто; дата редакции на /laws обязательна; мониторинг новых редакций — фаза 2 |

## 2. Эндпоинты

```
GET  https://cbd.minjust.gov.kg/api/v1/GetDocument?DocumentCode={code}
GET  https://cbd.minjust.gov.kg/api/v1/GetEdition?editionId={id}&lang={ru|kg}&exact=false
POST https://cbd.minjust.gov.kg/api/v1/GetDocuments?pageNumber=1&pageSize=12   // MVP не использует
```

## 3. Конфигурация корпуса (config/laws.json)

DocumentCode каждого закона находится руками через поиск на сайте (один раз), фиксируется вместе с ожидаемым названием — защита от опечатки в коде:

```json
[
  { "documentCode": "3-45", "expectedTitleContains": "Трудовой кодекс" },
  { "documentCode": "…",    "expectedTitleContains": "Семейный кодекс" }
]
```

## 4. C#-модели десериализации (Myizam.Ingestion/ApiModels/)

```csharp
// GetDocument
public sealed record GetDocumentResponse(
    DocumentStatus Status,
    string? DateAdopted,
    string? DateOfEntry,
    List<EditionInfo> Editions);

public sealed record DocumentStatus(string Code, string NameRus, string NameKyr);
// Действует: Code == "10", NameRus == "Действует"

public sealed record EditionInfo(
    long Id,
    int EditionCode,
    string NameRus,          // дата редакции строкой "dd.MM.yyyy"
    string? TextRusType,     // ".docx" — исходник Word, отсюда и HTML-каша
    string? TextKyrType);

// GetEdition
public sealed record GetEditionResponse(
    long Id,
    string NameRus,
    string? ContentRu,       // HTML; при lang=kg здесь КЫРГЫЗСКИЙ текст (!)
    string? ContentKg,       // легаси, всегда null — не использовать
    string? DateOfFutureEntry);
```

System.Text.Json, `PropertyNameCaseInsensitive = true`. Десериализация — с проверкой обязательных полей: null в Status или пустой Editions → исключение с documentCode в сообщении.

## 5. Пайплайн ingest (по каждому закону из конфига)

```
1. GET GetDocument(documentCode)
2. ПРОВЕРКИ (fail-fast, каждая падает с внятной ошибкой):
   a. status.NameRus == "Действует"        → иначе STOP «документ {code}: {status}»
   b. название содержит expectedTitleContains → защита от перепутанного кода
3. Выбор актуальной редакции:
   - распарсить NameRus каждой edition как дату (dd.MM.yyyy, ru-RU)
   - отбросить редакции с датой > сегодня
   - взять максимальную по дате; если существует будущая — warning в лог
4. GET GetEdition(editionId, lang=ru)
   - ContentRu null/пустой → STOP
   - если DateOfFutureEntry != null и <= сегодня — warning «возможно, есть более
     новая вступившая редакция» (перепроверить GetDocument)
5. Сохранить сырой HTML: data/raw/{documentCode}_{editionId}_ru.html (в .gitignore)
   — повторные прогоны парсера без повторных запросов к API (флаг --from-cache)
6. Парсинг HTML → дерево статей (см. §6)
7. Валидация дерева (см. §7)
8. Чанкинг (правила из ТЗ §4.4 без изменений: 1 статья = 1 чанк,
   контекстный заголовок, деление длинных по частям)
9. Эмбеддинги (батчи ≤100, идемпотентность по ContentHash) → запись в БД
   вместе с laws.status и laws.edition_date
```

Вежливость к API: задержка 1–2 с между запросами; осмысленный User-Agent
(`Myizam-Ingestion/1.0 (+github.com/erjigit7/myizam)`); суммарно на MVP ~14 запросов — нагрузки нет, но привычка правильная.

## 6. HTML-парсер (Parsers/MinjustHtmlParser.cs)

Библиотека: **AngleSharp** (устойчивее HtmlAgilityPack к Word-каше: вложенные span, mso-стили).

### 6.1 Предобработка

1. Распарсить документ; работать только с `<body>`
2. Удалить `<style>`, комментарии, `<o:p>` и прочие mso-огрызки
3. Для каждого блочного элемента (`p`, `div`, `tr`) получить **нормализованный текст**:
   - `&nbsp;` (U+00A0) → пробел; мягкие переносы (U+00AD) → удалить
   - схлопнуть повторные пробелы; Trim
4. Таблицы (`<table>`) конвертировать в текст построчно: ячейки через « | » — тарифные и прочие таблицы должны попасть в текст статьи, а не потеряться

### 6.2 Классификация абзацев — регулярки на НОРМАЛИЗОВАННОМ тексте

```csharp
// Якорь ^ обязателен: отсекает упоминания «...в соответствии со статьей 15»
// внутри текста (там строчная «с» и не начало абзаца)
static readonly Regex ArticleRx = new(
    @"^Статья\s+(\d+(?:-\d+)*)\s*\.?\s*(.*)$",
    RegexOptions.Compiled);

static readonly Regex ChapterRx = new(
    @"^Глава\s+(\d+(?:-\d+)*)\s*\.?\s*(.*)$",
    RegexOptions.Compiled);

static readonly Regex SectionRx = new(
    @"^Раздел\s+([IVXLC]+|\d+)\s*\.?\s*(.*)$",
    RegexOptions.Compiled);
```

- Захват `(\d+(?:-\d+)*)` покрывает «Статья 82», «Статья 82-1», «Статья 82-1-1» (нумерация вставленных статей)
- Заголовок статьи может быть в том же абзаце после номера ИЛИ отдельным следующим абзацем (Word-экспорт непредсказуем) → правило: если группа (2) пуста, а следующий абзац не матчится ни одним Rx и короче 200 символов — считать его заголовком
- Жирность/центрирование НЕ использовать как основной признак (Word кодирует её то `<b>`, то `style="font-weight:…"`, то классом) — только как вторичный сигнал в спорных случаях

### 6.3 Конечный автомат сборки дерева

```
состояние: текущие Section / Chapter / Article
для каждого блочного элемента:
  SectionRx  → закрыть текущую статью; новый Section
  ChapterRx  → закрыть текущую статью; новый Chapter
  ArticleRx  → закрыть текущую статью; новая Article(No, Title?)
  иначе      → если есть открытая Article — добавить текст к ней
               если Article ещё не было (преамбула, «Настоящий Кодекс…»)
                 → в Law.Preamble (индексируется отдельным чанком «Преамбула»)
```

### 6.4 Известные ловушки Word-HTML (проверить на реальном ТК в первую очередь)

- Примечания «(В редакции Закона КР от 23 июня 2026 года № …)» после заголовка статьи — ОСТАВЛЯТЬ в тексте статьи: юридически значимо и полезно в ответах
- Сноски/примечания в конце — прикрепляются к последней статье; проверить глазами, не мусор ли
- Разрыв абзаца внутри предложения (Word-артефакт) — не лечим: чанк = статья целиком, внутренние переносы безвредны
- «Статья» с заглавной в начале абзаца внутри цитаты другой нормы — редкий, но возможный ложный матч; ловится валидацией непрерывности номеров (§7)

## 7. Валидация после парсинга (отчёт в консоль + data/reports/{code}.md)

1. **Непрерывность номеров**: последовательность целых частей номеров статей возрастает; каждый пропуск — в отчёт (норма для «исключённых» статей, но каждый случай подтвердить глазами по сайту)
2. Нет пустых статей (номер есть, текста нет) — это почти всегда баг классификации заголовков
3. Сумма длин текстов статей ≥ 90% длины всего нормализованного текста body (детектор «потеряли кусок»)
4. Число статей правдоподобно (у кодекса — сотни; 12 статей у ТК = сломанный regex)
5. `--dry-run` печатает: дерево разделов/глав, число статей, 3 случайные статьи целиком — обязательный ручной просмотр перед боевым прогоном каждого нового закона

## 8. Юнит-тесты парсера (фикстуры — сохранённые фрагменты реального HTML в tests/fixtures/)

- статья с заголовком в той же строке / заголовком отдельным абзацем
- «Статья 82-1» и «82-1-1»
- упоминание «со статьей 15» внутри текста → НЕ новая статья
- таблица внутри статьи → текст с « | » сохранён
- преамбула до первой статьи → Law.Preamble
- nbsp/мягкие переносы нормализованы
- редакционное примечание после заголовка → в тексте статьи
